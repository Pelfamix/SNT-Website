Contexte

Log4Shell (CVE-2021-44228) est une vulnérabilité dans Log4j, une libraire de logging Java développée par Apache.
La vulnérabilité, qui existe sans être remarquée depuis 2013, a été divulguée en privé à Apache par un chercheur de l'équipe de sécurité d'Alibaba Cloud le 24 novembre 2021. Ils ont développé un patch pour pouvoir la solutionner, et a été révélée publiquement le 9 décembre 2021, lorsqu'Apache a sorti son patch 2.15.
Cependant, après quelques tests, il y avait encore des vulnérabilité qui subsistaient. Apache a sorti le patch 2.16, qui permet de régler, à priori, la quasi-totalité des vulnérabilités qui ont été découvertes à ce jour.


Impact

Le nombre de systèmes et le nombre d'applications qui sont impactés par cette vulnérabilité est très importante. Log4j est utilisée par énormément d'applications, et la quasi-totalité des versions de cette librairie est concernée (toutes les versions en dessous de 2.17). Ces applications peuvent être par exemple des programmes privés, mais peuvent aussi être des programmes commerciaux publics, notamment Minecraft: Java Edition, Steam, Amazon Web Services, etc.


Criticités et menaces

Cette vulnérabilité CVE-2021-44228 a été classé comme "critical", avec comme score de sévérité le maximum, 10/10. Cette faille permet une exécution de code arbitraire, c'est à dire l'éxécution de code à distance à une machine sans que le propriétaire soit d'accord.
Un attaquant peut donc faire du déni de service, ou bien upload un ransomware sur le serveur concerné, installer des crypto-miners, des bots, etc.


Explication de la vulnérabilité

Chaque fois que l'on trouve `${expression}`, les mécanismes de recherche Java trouvent la valeur de l'expression et la remplacent.
La recherche JNDI prend en charge des protocoles tels que LDAP, RMI, DNS et IIOP. Cependant, aucun contrôle n'est effectué sur les données récupérées à distance, et si le serveur distant est contrôlé par un attaquant, cela peut donner lieu à une injection de code, étant donné que c'est ensuite exécuté par le serveur.

Solutions

Malgré le fait que certains patchs existent déjà, il peut encore y avoir d'autres vulnérabilités présentes. La première chose à faire, est de continuer à s'informer au fur et à mesure.
De plus, Apache a déjà posté quelques articles pour contrer cette faille. La seule solution pour l'instant est de désactiver cette fonction de recherche JNDI. Il font donc soit mettre à jour Log4j vers les versions les plus récentes, soit désactiver manuellement la classe entière.
Une autre mesure à prendre est de mettre des filtres en place, au niveau des pare-feux par exemple, pour tenter de filtrer les requêtes HTTP qui exploitent la vulnérabilité (il est conseillé de considérer cette mesure comme complémentaire, car les filtrages qui se basent sur des patternes peuvent être contournés, et ne protègent pas entièrement le serveur).
